RESILIENCE TESTING
FOR EVERY ENGINEER.

Chaos engineering, made deterministic — no platform, no specialists. Just a binary and a YAML file you run on your laptop or in CI, anytime. Shinari brings up a real system, hits it with controlled faults — kill a process, partition the network, poison DNS — and keeps a living ledger of how it fails.
Every failure documented. Every fix detected.

Get started → Why a findings ledger? 
$ go build -o shinari ./cli && shinari run

A crash is a test case

The whole harness is one YAML file. Write the failure you fear on the left — Shinari runs it for real on the right.

scenarios/resilience/cache-outage.yml
kind: Scenario
name: checkout-survives-cache-outage

steadyState:            # only test a healthy system
  - run: http.get
    with: /health

method:
  - phase: "Kill the cache out from under the API"
    steps:
      - run: docker.kill
        with: redis
      - run: http.get   # checkout must answer without it
        with: /checkout/42
        as: rsp

  - phase: "Bring the cache back"
    steps:
      - run: docker.start
        with: redis

verify:
  - run: assert
    with: { of: "${rsp.total}", equals: 19.90 }
    desc: "served from Postgres, priced correctly"
  - run: http.get
    with: /metrics
    as: metrics
  - run: assert
    with: { of: "${metrics.p99_ms}", lt: 200 }
    desc: "p99 back under 200ms"
    finding: "cold cache — p99 spikes for ~30s after restart"
shinari run
$ shinari run

That finding: is the differentiator: a known gap stays documented, asserted, and green — the day someone fixes it, the run flips red and says promote this to a hard assertion.

What you can break

net.nxdomain

toxiproxy.latency

http.get · exec.run

probe real APIs, run any script
